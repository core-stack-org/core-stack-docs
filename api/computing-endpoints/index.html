    Computing API Endpoints¶

    The computing surface is defined in computing/urls.py and implemented in computing/api.py.

    Most of these handlers queue work and return a quick success response, while the heavy lifting happens asynchronously in pipeline modules.

    Current Behavior¶

    Many handlers call apply_async(...) on Celery tasks from computing/api.py, for example:

    • hydrology in generate_fortnightly_hydrology()
    • LULC in lulc_for_tehsil()
    • SWB in generate_swb()

    Two useful concrete patterns are visible in current code:

    try:
        state = request.data.get("state").lower()
        district = request.data.get("district").lower()
        block = request.data.get("block").lower()
        gee_account_id = request.data.get("gee_account_id")
        generate_facilities_proximity_task.apply_async(
            args=[state, district, block, gee_account_id], queue="nrm"
        )
        return Response({"Success": "Successfully initiated"})
    except Exception as exc:
        return Response({"Exception": exc})

    The second pattern is paired with a retry-aware Celery wrapper in generate_facilities_proximity_task(), which is documented further in Run Internal Tasks.

    Note

    Compute route protection is mixed in the current backend code. Some handlers use api_security_check(...), while others are plain @api_view(["POST"]) functions. Use Swagger and deployment policy as the final source for what is callable in a given environment.

    POST /api/v1/create_workspace/ create GeoServer workspace create_workspace()
    POST /api/v1/generate_block_layer/ generate admin boundary layer generate_admin_boundary()

    Hydrology and LULC¶

    Route Purpose Source
    POST /api/v1/hydrology_fortnightly/ fortnightly hydrology task generate_fortnightly_hydrology()
    POST /api/v1/hydrology_annual/ annual hydrology task generate_annual_hydrology()
    POST /api/v1/lulc_for_tehsil/ LULC v2 or v3 by block or tehsil lulc_for_tehsil()
    POST /api/v1/lulc_v2_river_basin/ basin-scale LULC v2 lulc_v2_river_basin()
    POST /api/v1/lulc_v3_river_basin/ basin-scale LULC v3 lulc_v3_river_basin()

    POST /api/v1/lulc_vector/ vectorized LULC output lulc_vector()
    POST /api/v1/lulc_v4/ newer LULC task path lulc_v4()

    Derived Layers¶

    Route Purpose Source
    POST /api/v1/generate_ci_layer/ cropping intensity generate_ci_layer()
    POST /api/v1/generate_swb/ surface water bodies generate_swb()
    POST /api/v1/generate_drought_layer/ drought layer generate_drought_layer()
    POST /api/v1/generate_terrain_descriptor/ terrain clusters generate_terrain_descriptor()
    POST /api/v1/generate_terrain_raster/ terrain raster task generate_terrain_raster()

    There are many more compute routes in computing/urls.py, including stream order, restoration opportunity, aquifer vectors, ponds, wells, NDVI time series, slope percentage, and facilities proximity.

    Block Enrichment and Clip Routes¶

    Route Purpose Source
    POST /api/v1/generate_lcw/ queue Land Conflict Watch clipping for a block generate_lcw()
    POST /api/v1/generate_facilities_proximity/ queue facilities distance and enrichment workflow generate_facilities_proximity()

    These are good references when you want to expose a new block-scoped computation through an HTTP entry point without forcing contributors to begin from Django internals.

    What Happens After Submission¶

    The common shape of the compute flow is:

    1. route declared in computing/urls.py
    2. request handled in computing/api.py
    3. task queued with Celery
    4. pipeline module runs
    5. results are pushed toward GeoServer or stored metadata

    The publication helper layer is in computing/utils.py.
