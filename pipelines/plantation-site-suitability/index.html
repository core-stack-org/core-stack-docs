    Plantation Site Suitability¶

    This workflow generates plantation site-suitability outputs from either:

    • a project ROI built from uploaded KML files
    • an admin-unit ROI built from an existing MWS layer

    In the current backend, the project-backed path is the fuller and safer workflow to document and test first.

    Primary sources:

    • computing/api.py
    • computing/plantation/site_suitability.py
    • computing/plantation/site_suitability_vector.py
    • computing/plantation/site_suitability_raster.py
    • computing/plantation/utils/process_profile.py
    • plantations/models.py

    What The Workflow Produces¶

    The current code can produce:

    • a suitability raster with weighted Climate, Soil, Topography, Ecology, Socioeconomic, and final-score bands
    • a suitability vector for project sites, enriched with patch score, patch suitability, confidence, NDVI, LULC, and site-property fields
    • GeoServer-published outputs in the plantation workspace

    Project-backed outputs land under the plantation GEE path:

    projects/ee-corestackdev/assets/apps/plantation/<organization>/<project>/

    Admin-unit-backed raster outputs land under the MWS path:

    projects/ee-corestackdev/assets/apps/mws/<state>/<district>/<block>/

    Route And Request Shape¶

    The computing route is:

    POST /api/v1/plantation_site_suitability/

    Project-backed request¶

    {
  "project_id": 12,
  "start_year": 2021,
  "end_year": 2023,
  "gee_account_id": 1
}

    Admin-unit-backed request¶

    {
  "state": "bihar",
  "district": "nalanda",
  "block": "hilsa",
  "start_year": 2021,
  "end_year": 2023,
  "gee_account_id": 1
}

    Note

    The route accepts both request shapes, but the current backend only continues into full vector generation for the project-backed path. The non-project path still goes through raster generation in get_pss(...), then returns early from check_site_suitability(...).

    Required DB Setup For Project-Backed Runs¶

    This is the missing setup that tends to break fresh-install tests.

    1. Create an organization¶

    The Project model requires an Organization, and the plantation task uses project.organization.name to build the Earth Engine asset path.

    2. Create a project linked to that organization¶

    For this workflow, the project should be:

    • linked to the organization above
    • app_type=plantation
    • enabled=True
    • assigned to a valid state

    The project_id you pass to site_suitability() comes from this record.

    3. Upload at least one KML file for that project¶

    The project path builds the ROI by reading KMLFile records and combining them in combine_kmls(...).

    Without project KMLs, the ROI assembly step has nothing to merge and the workflow cannot build a project footprint.

    4. Create a plantation profile for that project¶

    The raster stage reads plantation profile configuration from the database.

    Current code behavior is slightly stricter than it first appears:

    • get_pss(...) always loads PlantationProfile.objects.get(profile_id=1) as a default baseline
    • if a project-specific PlantationProfile exists, its weights and variables are merged on top of that baseline

    On a fresh database, the safest setup is:

    1. create the plantation profile for the project before the first API run
    2. make sure at least one plantation profile exists at all, so the default profile_id=1 lookup has something to load

    If you are doing this manually, Django admin is a straightforward path. The backend also exposes nested project-scoped plantation KML and plantation profile endpoints in plantations/urls.py.

    Year Range: The Important Constraint¶

    The plantation route accepts both start_year and end_year, but the main raster computation does not use the request's start_year directly.

    Inside check_site_suitability(...), the raster stage calls:

    get_pss(..., start_year=end_year - 2, end_year=end_year, ...)

    So the effective raster window is always a three-year span anchored on end_year.

    This matters because the plantation helpers read the pan-India LULC dataset as:

    .../pan_india_lulc_v3_<year>_<year+1>

    The practical consequence is:

    • minimum safe end_year: 2019
    • why: 2019 resolves to a raster window starting at 2017
    • risk: 2018 or lower resolves to 2016 internally, and that LULC asset path does not exist in the expected dataset window

    Warning

    If you pass end_year=2018 or lower, expect a GEE asset-not-found error from the LULC-dependent portion of the plantation workflow.

    Code Path¶

    The current project-backed execution sequence is:

    1. computing/api.py queues site_suitability.apply_async(...)
    2. computing/plantation/site_suitability.py initializes GEE and loads the project
    3. the task fetches project KMLs, creates or updates the project ROI asset in Earth Engine, and loads that ROI as a FeatureCollection
    4. check_site_suitability(...) calls get_pss(...) to build the weighted suitability raster
    5. the project path continues into vector generation, attaching NDVI, LULC, and site-property summaries to each feature
    6. outputs are exported to Earth Engine and then synced toward GeoServer

    At a high level, the weighted raster combines:

    • Climate
    • Soil
    • Topography
    • Ecology
    • Socioeconomic factors

    The profile configuration that drives those weights is built in process_project_profile(...) and stored in PlantationProfile.

    Operational Notes¶

    Large project ROIs use the helper account path¶

    If the ROI contains more than 50 features, site_suitability_vector.py switches to chunked vector generation.

    That path uses:

    • gee_account.helper_account
    • helper asset folders for chunk export and merge

    If the GEEAccount was created by the installer import path, it is usually self-linked by default. If you created it manually and left helper_account blank, very large project runs can fail here.

    Plantation outputs also depend on GCS¶

    The raster publication step calls:

    • sync_raster_to_gcs(...)
    • sync_raster_gcs_to_geoserver(...)

    So plantation site suitability is not only a GEE workflow. It also depends on working Google Cloud Storage and GeoServer configuration.

    See:

    • Google Cloud Storage
    • Google Earth Engine
    • GeoServer

    The non-project path is currently partial¶

    The request shape with state, district, and block is still useful for raster generation. But as of the current code, check_site_suitability(...) returns early when project is absent, so it does not continue into the full vector export branch.

    Treat project-backed execution as the main documented workflow until that behavior changes.
